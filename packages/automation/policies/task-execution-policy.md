# Task Execution Policy

Defines how tasks move from idea to backlog, queue, implementation, review, and done.

## Rules

1. Prefer explicit records over vague memory.
2. Keep work project-scoped whenever possible.
3. Mark risky work before generating execution prompts.
4. Require approval for destructive, financial, external, or production actions.
5. Update project memory and documentation after meaningful changes.
6. Record decisions when architecture, product direction, provider choice, or trading risk changes.

## Human approval required for

- Production deployment.
- External communication.
- Account, billing, or provider changes.
- Data deletion or migration.
- Trading, investing, or financial order execution.
- Secrets or credential changes.
